\chapter{Methodology}

\section{Current State and Proposed Solution}

Getting data on resources consumption in the shared housing sector it's currently done in four ways: using AMI systems, 
manual readings from electric, water and gas meters, comparing utility bills, and private smart monitor devices. None of
the studies found in the literature review mentioned the use of open smart monitoring systems, an standardised API, and 
the benefits of cloud computing.

The proposed solution generates a daily report for guests delivered via text message (AirBnB's chat). The report will 
have information on the number of KiloWatts, and liters of water used, and the average temperature of the room in the 
last 24 hours. It will compare these figures with the results from the day before (if available). 

The system will also generate an end-of-stay report for the guest. This report will have the same information as the 
daily report, but it will also include the total amount of KiloWatts and liters of water used during the stay. It will
highlight improvements or deteriorations in the consumption of resources.

As for the hosts, a dashboard is available on the web showing historical data in form of graphs to show electricity and
water consumption, and the temperature of the room and outside. The final report from the user is also forwarded to the 
host.

For external users, the API has a endpoint for downloading datasets in CSV format that can be shared with third parties
for further analysis and research.

\subsection{Requirements}
The following requirements improve the current state of solutions for monitoring resources consumption in 
shared housing:

\begin{enumerate}
    \item \textbf{Fine-grained readings:} The sensors should read at a minimum of 2 seconds to 
    enable near real-time data logging of temperature, humidity, water consumption.
    \item \textbf{High frequency telemetry:} The sensors should be able to send data to the cloud at a minimum of every 
    2 minutes.
    \item \textbf{Continous connectivity:} The sensors should be able to maintain a connection to the cloud at all 
    times. 
    \item \textbf{Secure data transmission:} Authentication and encryption should be used to ensure data integrity and
    privacy.
    \item \textbf{Data visualization:} The system should be able to visualize the data in real time, as well as the 
    reports. This all should be done in a user-friendly way.
    \item \textbf{User authentication:} The system should have a user authentication system to prevent unauthorized 
    access.
    \item \textbf{Reports generation:} The system should be able to generate daily and end-of-stay reports for guests 
    in a comprehensible user friendly way.
    \item \textbf{Open API standard:} The API should comply with the Open API standard to ensure interoperability with 
    other systems.
    \item \textbf{Scalabitlity and maintainability:} The system should be able to handle a large number of sensors and 
    users. It should also be easy to maintain.
\end{enumerate}

The general system architecture of the proposed solution is shown in Figure \ref{fig:architecture-diagram}. The solution 
consists of four main layers: device, data, communication, and presentation. There are two main components, the smart
monitoring system and the web application that runs on the cloud. In the next sections, each layer and component will be
explained in detail.

\subsection{Device Layer}

\subsubsection{Microcontrollers}

It consists of three Espressif ESP32-DevKitC \cite{Espressif2024} microcontrollers (pictured in 
\ref{fig:connectivity-diagram}) connected to the sensors. One measures energy consumption, the other water usage and 
the third one indoor and outdoor temperatures. This development board was chosen for its extensive documentation, 
integrated WiFi and compatibility with the peripherals used. The idea was to have a low cost solution with fast 
development and deployment times.

FreeRTOS \cite{FreeRTOS2024} runs on top of these microcontrollers to allow a secure and resource aware solution for 
these embedded systems (microcontrollers). This was chosen as opposed to running the code on top of the chip because it 
allows to run multiple tasks at the same time with a predictible execution pattern, which is ideal for sensing and 
keeping connection with the internet alive at the same time (a requirement for this project).

The microcontrollers where programed in C++ using PlatformIO IDE (Integrated Development Environment) 
\cite{PlatformIO2024} for embedded systems. This development environment was chosen because of its compatibility with 
the microcontrollers and the sensors chosen, integration with thousands of libraries and collaboration principles.

\subsubsection{Sensors}

The interaction with the sensors was done using Arduino \cite{Arduino2024a} libraries, as these are the most used and 
well documented. They have been tested by a lot of projects (even in commercial products) and have an active community. 
The sensors used were the following:

\begin{itemize}
    \item \textbf{Energy usage:} Non-invasive AC Current Sensor SCT-013 \cite{CRCibernetica2024b} connected to 
    the ESP32 Energy Monitor Shield \cite{CRCibernetica2024c}.
    \item \textbf{Water consumption:} Liquid Flow Meter - Plastic 1/2" NPS Threaded TF-S201 \cite{CRCibernetica2024d}.
    \item \textbf{Temperature:} DS18B20 Temperature Sensor \cite{CRCibernetica2024e}.
\end{itemize}

Communication between the device layer and the data layer is done using the MQTT (Message Queuing Telemetry Transport) 
\cite{MQTT2024} protocol. MQTT was chosen because it is lightweight, secure and uses minimal network bandwidth. It also 
has become a standard in IoT for connecting remote devices.

\subsection{Data Layer}

Two separate databases are kept, a time series database for every reading from the sensors and a non-relational database 
for user and listing data, as well as device metadata. For the time series database the open source ThingsBoard IoT 
Platform \cite{ThingsBoard2024}was selected because it has integrated data collection and device management, which makes 
it ideal to collect data from sensor at big scale as opposed to an in house solution. The platform also has a well 
documented API for interaction with the devices and the data. 

\subsection{Communication Layer}

It is a REST API that interconnects the data layer with the application layer. It works as a way to have access to user, 
listing, and device (including time series) data. It complies with Open API \cite{OpenAPI2024} specification for 
standardization. This layer allows clients (applications) to access information from the data layer in a secure and 
controlled way.

\subsubsection{Communication with Airbnb}

Currently AirBnB does not have a public API, it provides a private API for authorized partners \cite{Airbnb2024c}. 
Becoming a partner was not possible for this project as there is no official way to apply for this. Rather, an AirBnB
manager contacts potential external partners. The solution was to simulate responses from the API using a web browser
to extract the payload from network activity on an AirBnB account using the web application.

\subsection{Presentation Layer (User Interface)}

A User Interface (UI) that can be used from personal computers, tablets and cellphones. It gives the host access to
a dashboard with in-depth real-time data of every listing that shows energy and water consumption, and temperature.

Figure \ref{fig:connectivity-diagram} shows a high-level diagram of the flow of data between the different components

\section{Implementation}

\subsection{Sensing Nodes}

Three types of sensing nodes were developed for the project: power, water flow, and temperature. Each node is based on 
the ESP32 microcontroller and uses different sensors to measure the desired parameters.

\subsubsection{Power Node}

The power node is by far the most complex of the three nodes. It uses two current sensors attached to each phase of the 
120V AC power line, a small circuit was built to adapt the current so it could be measured by the ADC (Analog to Digital
Converter) input of the microcontroller. This circuit consists of a voltage divider used to remove the negative part of 
the sine wave combined with a capacitor to stabilize the signal, and a resistor to convert the current to voltage. The 
calculations for the resistors and capacitors were done using Arduino's EmonLib documentation \cite{Arduino2024b}.

At the begining, the ESP32 was able to interact with these sensors and obtain the data with no problems, but once the 
full program was loaded it kept getting stuck due to memory and CPU usage. Several optimizations were tried, like 
minimizing the use of variables, removing all log messages or simplifying tasks but it was simply not possible to fix 
this behaviour. Ultimately, an Arduino Leonardo was connected to the circuit and a script that read the data from the
sensors was deployed on it, then the ESP32 was attached to the Leonardo via UART (Universal Asysnchronous Receiver
Transmitter) and the reading thread running on the ESP32 originally was replaced by a thread that reads from the serial
ports. This solution fixed the problem and the node was able to monitor without any issues. Figure 
\ref{fig:energy-node-schematic} shows the schematic of the power node.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.8\textwidth]{energy-node-schematic.png}
    \caption{Power node schematic}
    \label{fig:energy-node-schematic}
\end{figure}

Once the current was read, EmonLib was used to calculate the correct power by multiplying the current by the fixed
voltage (120V). Since the data was collected every second the power over time (energy) was calculated by adding the 
power every second for a minute and then dividing by 60 to get watt-seconds or Joules (this calculation can be seen in 
equation \ref{eq:energy-calculation}). After the energy was calculated, it was sent to the cloud every minute. 

\begin{equation}
    \label{eq:energy-calculation}
    \text{Energy} = \frac{1}{60} \sum_{i=1}^{60} (I_i \times V_i)
\end{equation}

\subsubsection{Water Flow Node}

This node uses a water flow sensor to measure the flow rate. This sensor uses Hall effect to generate a pulse every time
some water passes through it. The program counts the number of pulses every second and then it multiplies that by a 
calibration factor (7.11 in this case) to calculate the flow rate, this is then divided by 1000 to get the number of
liters that passed during that second. The sensor was connected to the ESP32 as shown in Figure 
\ref{fig:water-flow-node-schematic}.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.5\textwidth]{water-flow-node-schematic.png}
    \caption{Water flow node schematic}
    \label{fig:water-flow-node-schematic}
\end{figure}

Using equation \ref{eq:water-volume-calculation} the volume of water passed in the last minute was calculated and then 
sent to the database.

\begin{align}
    \label{eq:water-volume-calculation}
    \text{Volume} &{}= \sum_{i=1}^{60} Q_i \times \underbrace{\Delta t}_{=1\text{s}}\\
                  &{}= \sum_{i=1}^{60} Q_i
\end{align}

\subsubsection{Temperature Node}

The temperature sensors were connected to the ESP32 as shown in Figure \ref{fig:temp-node-schematic}. These sensors were
controlled using the OneWire library \cite{Arduino2024c}. The temperature was read every 2 seconds for 60 seconds and 
then it was divided by 30 to get the average temperature in the past minute. Equation 
\ref{eq:temperature-calculation} depicts the mathematical representation of this. After the average temperature was 
calculated, it was sent to the database.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.4\textwidth]{temp-node-schematic.png}
    \caption{Temperature node schematic}
    \label{fig:temp-node-schematic}
\end{figure}

\begin{equation}
    \label{eq:temperature-calculation}
    \text{Average Temperature} = \frac{1}{30} \sum_{i=1}^{30} T_i
\end{equation}

\subsubsection{Timeseries Database}

ThingsBoard's internal PostgrepSQL \cite{PostgreSQL2024} was used as the timeseries database for storing historical data

\subsection{REST API}

This API was designed to manage users, providers, property listings, and devices for a hospitality platform like 
AirBnB, where properties are listed and managed by users. It enables seamless integration with third-party providers 
like AirBnB, VRBO, and HomeAway, and facilitates the management of property data from smart devices.

\subsubsection{Users}

The API includes a user management system. Users can register by providing essential information such as their username, 
email, and password. Upon successful registration, the system confirms that the user has been created. Once registered, 
users can log in by providing their credentials (username and password), which are authenticated by the system. If 
successful, the user is provided with an authentication token that must be used in subsequent requests to ensure 
security. Each user has a unique ID, which is tied to their providers, listings, and jobs.

In addition to managing personal information, users can also sync their property listings with third-party providers 
such as Airbnb or HomeAway. This ensures that all the properties the user has listed on these platforms are 
automatically updated in the system.

\subsubsection{Providers}
Providers are third-party platforms like AirBnB, VRBO \cite{VRBO2024}, or Booking.com \cite{Booking2024}, where 
properties are listed. The API allows users to create and manage these provider connections. When a user adds a new 
provider, the system stores the provider information, linking it to the user.

\subsubsection{Listings}

Listings represent the properties users manage on the platform. The API provides functionality to import listings, 
once listings are imported, they can be viewed as a list or individually by their unique ID. Users can fetch all their 
listings to see which properties they are currently managing. Additionally, the system allows for enabling or disabling 
listings, giving users full control over which properties are being monitored.

\subsubsection{Reservations}

Just as Listings, Reservations are imported from the provider. They are used internally to calculate the daily and 
end-of-stay consumption reports. The API allows for fetching all reservations for a listing, as well as individual
reservations by their unique ID.

\subsubsection{Devices}

These devices are linked to specific listings and can be managed through the API. The API supports adding new devices by 
providing details like the type of device (e.g., energy meter), its unique identifier, and the token that authenticates 
the device with ThingsBoard. Users can retrieve a list of all devices associated with their listings. If a user wants to 
see detailed information about a particular device, they can fetch it using the device's unique ID. The API also 
supports updating device information or removing devices from the system when they are no longer needed.

\subsubsection{Reports}

Reports are generated daily and at the end of the guest's stay. The API provides endpoints to fetch these reports for a
specific listing.

\subsubsection{Agenda Jobs}

Two types of jobs were implemented, one that runs every two minutes to import the listings and reservations from the 
provider and another one that runs every days at 2pm and calculates the consumption in the last 24 hours or the total
one.

\subsubsection{Schema Components}

The API uses specific schemas to define the structure of data objects exchanged between the client and server. For 
example, the User schema defines fields like username, email, and password that are required when registering a user. 
The Provider schema defines the fields necessary for creating a provider, such as the provider name and the associated 
user ID. Similarly, the Listing schema outlines the structure for properties, including details like title, location, 
and provider. Each device follows a schema as well, which includes fields for the device type, its unique ID, the token 
for authentication, and the associated listing. This consistent structure allows for easy integration and ensures data 
integrity.

In addition, the API provides a standardized ApiResponse schema to maintain consistent error and success messages across 
different endpoints. Whether it's creating a user or fetching device details, the response messages provide clarity on 
the status of the operation.

\subsection{Continuous Integration}

A continuous integration pipeline was set up to automate the build and test processes. The pipeline consists of several 
stages, each responsible for a specific task. The pipeline is triggered whenever changes are pushed to the repository, 
ensuring that the code is always up-to-date and tested.

\subsubsection{Firmware}

The code is built using the PlatformIO build system, which compiles the firmware for the sensing nodes. Errors in the 
code would be shown in the logs and the build would fail. An example run of the pipeline can be seen in Figure
\ref{fig:firmware-build}.

\subsubsection{API}

Just like with the firmware the API is built and the unit and integration tests are run to ensure that the changes made 
to the code do not break existing functionality. Figure \ref{fig:api-build} shows an example run of the pipeline.

\subsubsection{Development Environment}

The development environment was created using Docker Compose \cite{Docker2024}, which allowed for easy setup and 
teardown. The environment consisted of a MongoDB container, a ThingsBoard container, a NodeJS (API) container, and an 
Appsmith container, Docker Compose was used to configure all the containers and services in a single file. This made it
easy to start and stop the environment with a single command.

\subsubsection{Deployment}

The complete solution was deployed on virtual machine hosted on Google Cloud. The virtual machine was configured with
Docker and Docker Compose to run the development environment, but with a slighty different configuration. The process of
deploying the project was to SSH into the virtual machine, update the repository, and build or rebuild the containers.
The original idea was to automate this process with a CI/CD pipeline but it was not possible due to time constraints.
