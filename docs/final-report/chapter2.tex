\chapter{Methodology}

\section{Current State and Proposed Solution}

Getting data on resources consumption in the shared housing sector it's currently done in four ways: using AMI systems, 
manual readings from electric, water and gas meters, comparing utility bills, and private smart monitor devices. None of
the studies found in the literature review mentioned the use of open smart monitoring systems, an standardised API, and 
the benefits of cloud computing.

The proposed solution generates a daily report for guests delivered via text message (AirBnB's chat). The report will 
have information on the number of KiloWatts and liters of water used, and the average temperature of the room in the 
last 24 hours. It will compare this figures with the results from the day before (if available). 

The system will also generate an end-of-stay report for the guest. This report will have the same information as the 
daily report, but it will also include the total amount of KiloWatts and liters of water used during the stay. It will
highlight and improvements or deteriorations in the consumption of resources.

As for the hosts, a dashboard is available on the web showing historical data in form of graphs to show electricity and
water consumption, and the temperature of the room and outside. The final report from the user is also forwarded to the 
host.

For external users, the API has a endpoint for downloading datasets in CSV format that can be shared with third parties
for further analysis and research.

\subsection{Requirements}

\begin{enumerate}
    \item \textbf{Fine-grained readings:} The sensors should read at a minimum of 2 seconds to 
    enable near real-time data logging of temperature, humidity, water consumption.
    \item \textbf{High frequency telemetry:} The sensors should be able to send data to the cloud at a minimum of every 
    2 minutes.
    \item \textbf{Continous connectivity:} The sensors should be able to maintain a connection to the cloud at all 
    times. 
    \item \textbf{Secure data transmission:} Authentication and encryption should be used to ensure data integrity and
    privacy.
    \item \textbf{Data visualization:} The system should be able to visualize the data in real time, as well as the 
    the reports. This all should be done in a user-friendly way.
    \item \textbf{User authentication:} The system should have a user authentication system to prevent unauthorized 
    access.
    \item \textbf{Reports generation:} The system should be able to generate daily and end-of-stay reports for guests.
    \item \textbf{Open API standard:} The API should comply with the Open API standard to ensure interoperability with 
    other systems.
    \item \textbf{Scalabitlity and maintainability:} The system should be able to handle a large number of sensors and 
    users. It should also be easy to maintain.
\end{enumerate}

The general system architecture of the proposed solution is shown in Figure \ref{fig:architecture-diagram}. The solution 
consists of four main layers: device, data, communication, and presentation. There are two main components, the smart
monitoring system and the web application that runs on the cloud.

\subsection{Device Layer}

\subsubsection{Microcontrollers}

It consists of three Espressif ESP32-DevKitC \cite{Espressif2024} microcontrollers (pictured in 
\ref{fig:connectivity-diagram}) connected to the sensors. One measures water consumption, the other energy usage and 
the third one indoor and outdoor temperatures. This development board was chosen for it's extensive documentation, 
integrated WiFi and compatibility with the peripherals used. The idea was to have a low cost solution with fast 
development and deployment times.

FreeRTOS \cite{FreeRTOS2024} runs on top of these microcontrollers to allow a secure and resource aware solution for 
these embedded systems (microcontrollers). This was chosen as opposed to running the code on top of the chip because it 
allows to run multiple tasks at the same time with a predictible execution pattern, which is ideal for sensing and 
keeping connection with the internet alive at the same time (a requirement for this project).

The microcontrollers where programed in C++ using PlatformIO IDE (Integrated Development Environment) for embedded
systems. This development environment was chosen because of its compatibility with the microcontrollers and the sensors 
chosen, its focus on test-driven methodologies and collaboration principles.

\subsubsection{Sensors}

The interaction with the sensors was done using Arduino \cite{Arduino2024a} libraries, as these are the most common and 
well documented. They have been tested by a lot of projects (even in commercial products) and have an active community. 
The sensors used are the following:

\begin{itemize}
    \item Energy usage: Non-invasive AC Current Sensor SCT-013 \cite{CRCibernetica2024b} connected to 
    the ESP32 Energy Monitor Shield \cite{CRCibernetica2024c} and controlled by EmonLib \cite{Arduino2024b} library.
    \item Water consumption: Liquid Flow Meter - Plastic 1/2" NPS Threaded TF-S201 \cite{CRCibernetica2024d}.
    \item Temperature: DS18B20 Temperature Sensor \cite{CRCibernetica2024e} controlled by DallasTemperature 
    \cite{Arduino2024c} library.
\end{itemize}

Communication between the device layer and the data layer is done using the MQTT (Message Queuing Telemetry Transport) 
\cite{MQTT2024} protocol. MQTT was chosen because it is lightweight, secure and uses minimal network bandwidth. It also 
has become a standard in IoT for connecting remote devices.

\subsection{Data Layer}

Two separate databases are kept, a time series database for every reading from the sensors and a non-relational database 
for user and listing data, as well as device metadata. For the time series database the open source ThingsBoard IoT 
Platform \cite{ThingsBoard2024}was selected because it has integrated data collection and device management, which makes 
it ideal to collect data from sensor at big scale as opposed to an in house solution. The platform also has a well 
documented API for interaction with the devices and the data. 

\subsection{Communication Layer}

It is a REST API that interconnects the data layer with the application layer. It works as a way to have access to user, 
listing, and device (including time series) data. It complies with Open API \cite{OpenAPI2024} specification for 
standardization. This layer allows clients (applications) to access information from the data layer in a secure and 
controlled way.

\subsubsection{Communication with Airbnb}

Currently AirBnB does not have a public API, it provides a private API for authorized partners \cite{Airbnb2024c}. 
Becoming a partner was not possible for this project as there is no official way to apply for this. Rather, an AirBnB
manager contacts potential external partners. The solution was to simulate responses from the API using a web browser
to extract the payload from network activity on an AirBnB account using the web application.

\subsection{Presentation Layer (User Interface)}

A User Interface (UI) that can be used from personal computers, tablets and cellphones. It gives the host access to
a dashboard with in-depth real-time data of every listing that shows energy and water consumption, and temperature. It 
also provides historical data on previous stays organized by listings and guests.

\subsection{Data Flow}
Figure \ref{fig:connectivity-diagram} shows a high-level diagram of the flow of data between the different components 
of the system. The microcontrollers send data to the ThingsBoard IoT Platform through an MQTT connection, which is then
stored in a time series database. The client application uses the API to access the data and present it to the host. 
The API and databases are hosted on the Google Cloud \cite{GoogleCloud2024} for security, scalability and reliability.

\section{Implementation}

\subsection{Sensing Nodes}

Three types of sensing nodes were developed for the project: power, water flow, and temperature. Each node is based on 
the ESP32 microcontroller and uses different sensors to measure the desired parameters.

\subsubsection{Power Node}

Energy consumed was calculated by summing the power every second for 60 seconds, and then dividing by 60:

\begin{equation}
    \text{Energy} = \frac{1}{60} \sum_{i=1}^{60} (I_i \times V_i)
\end{equation}

This gives the total energy used in that minute (in watt-seconds or Joules).

\begin{figure}[!ht]
    \centering

    \caption{Power node schematic}

\end{figure}

\subsubsection{Water Flow Node}

The water flow node uses a flow sensor to measure the flow rate of water. The flow sensor outputs a pulse every time a
certain volume of water passes through it. The volume of water is calculated by counting the number of pulses over a 
period of one second. The volume is then uploaded to the cloud.

\begin{align}
    \text{Volume} &{}= \sum_{i=1}^{60} Q_i \times \underbrace{\Delta t}_{=1\text{s}}\\
                  &{}= \sum_{i=1}^{60} Q_i
\end{align}

Since we are reading every second, it just adds the values to get the total volume.

\subsubsection{Temperature Node}

Calculates the average temperature over 120 seconds (reading every two seconds), then uploads it.

\begin{equation}
    \text{Average Temperature} = \frac{1}{30} \sum_{i=1}^{120} T_i
\end{equation}

\subsection{REST API}

This API is designed to manage users, providers, property listings, and devices for a platform similar to Airbnb, where properties are listed and managed by users. It enables seamless integration with third-party providers like Airbnb, VRBO, and HomeAway, and facilitates the management of property data along with smart devices (such as energy or water usage sensors) associated with those properties.

\subsubsection{Users}

The API includes a comprehensive user management system. Users can register by providing essential information such as their username, email, and password. Upon successful registration, the system confirms that the user has been created. If the user already exists, the API will return an error.
Once registered, users can log in by providing their credentials (username and password), which are authenticated by the system. If successful, the user is provided with an authentication token that must be used in subsequent requests to ensure security. If login credentials are incorrect, the system will reject the login attempt, returning an error message.
Users can also view a list of all registered users in the system. This operation requires proper authentication and is likely restricted to administrative users. Each user has a unique ID, which can be used to fetch more detailed information about the user, such as their profile details. Updates to a user’s profile, like changing their email address or name, can be done by sending a patch request to the system. Similarly, users can be deleted from the system if necessary.

In addition to managing personal information, users can also sync their property listings with third-party providers such as Airbnb or HomeAway. This ensures that all the properties the user has listed on these platforms are automatically updated in the system.
Providers
Providers are third-party platforms like Airbnb, VRBO, or HomeAway, where properties are listed. The API allows users to create and manage these provider connections. When a user adds a new provider, the system stores the provider information, linking it to the user. If the provider already exists for the user, the system returns an error to prevent duplication.
Providers can be deleted as well if the user decides to remove the connection. For example, if a user no longer wants to sync their Airbnb listings, they can delete the Airbnb provider from their account.

\subsubsection{Listings}

Listings represent the properties users manage on the platform. These properties are usually listed on third-party services like Airbnb or VRBO and can be synced with the system. The API provides functionality to create new listings, where users input details like the title, description, location, and the provider (such as Airbnb) the listing is associated with.
Once listings are created, they can be viewed as a list or individually by their unique ID. Users can fetch all their listings to see which properties they are currently managing. Additionally, the system allows for editing or deleting listings, giving users full control over their property data.
There is also the capability to enable or disable listings. Enabling a listing marks it as active, making it visible to potential renters. Conversely, disabling a listing temporarily deactivates it, which might be useful if the property is undergoing maintenance or unavailable for some time.

\subsubsection{Devices}

Devices refer to the hardware installed at a property to monitor usage or conditions, such as sensors that track energy consumption, water usage, or temperature. These devices are tied to specific listings and can be managed through the API.
The API supports adding new devices by providing details like the type of device (e.g., energy meter), its unique identifier, and the token that authenticates the device. Each device is associated with a listing so that users can monitor specific properties.
Users can retrieve a list of all devices associated with their listings. If a user wants to see detailed information about a particular device, they can fetch it using the device’s unique ID. The API also supports updating device information or removing devices from the system when they are no longer needed.

\subsubsection{Schema Components}

The API uses specific schemas to define the structure of data objects exchanged between the client and server. For example, the User schema defines fields like username, email, and password that are required when registering a user. The Provider schema defines the fields necessary for creating a provider, such as the provider name and the associated user ID. Similarly, the Listing schema outlines the structure for properties, including details like title, location, and provider.
Each device follows a schema as well, which includes fields for the device type, its unique ID, the token for authentication, and the associated listing. This consistent structure allows for easy integration and ensures data integrity.
In addition, the API provides a standardized ApiResponse schema to maintain consistent error and success messages across different endpoints. Whether it's creating a user or fetching device details, the response messages provide clarity on the status of the operation.
