\chapter{Methodology}

\section{Current State and Proposed Solution}

Getting data on resources consumption in the shared housing sector it's currently done in four ways: using AMI systems, 
manual readings from electric, water and gas meters, comparing utility bills, and private smart monitor devices. None of
the studies found in the literature review mentioned the use of open smart monitoring systems, an standardised API, and 
the benefits of cloud computing.

The proposed solution generates a daily report for guests delivered via text message (AirBnB's chat). The report will 
have information on the number of KiloWatts, and liters of water used, and the average temperature of the room in the 
last 24 hours. It will compare these figures with the results from the day before (if available). 

The system will also generate an end-of-stay report for the guest. This report will have the same information as the 
daily report, but it will also include the total amount of KiloWatts and liters of water used during the stay. It will
highlight improvements or deteriorations in the consumption of resources.

As for the hosts, a dashboard is available on the web showing historical data in form of graphs to show electricity and
water consumption, and the temperature of the room and outside. The final report from the user is also forwarded to the 
host.

For external users, the API has a endpoint for downloading datasets in CSV format that can be shared with third parties
for further analysis and research.

\subsection{Requirements}
The following requirements improve the current state of solutions for monitoring resources consumption in 
shared housing:

\begin{enumerate}
    \item \textbf{Fine-grained readings:} The sensors should read at a minimum of 2 seconds to 
    enable near real-time data logging of temperature, humidity, water consumption.
    \item \textbf{High frequency telemetry:} The sensors should be able to send data to the cloud at a minimum of every 
    2 minutes.
    \item \textbf{Continous connectivity:} The sensors should be able to maintain a connection to the cloud at all 
    times. 
    \item \textbf{Secure data transmission:} Authentication and encryption should be used to ensure data integrity and
    privacy.
    \item \textbf{Data visualization:} The system should be able to visualize the data in real time, as well as the 
    reports. This all should be done in a user-friendly way.
    \item \textbf{User authentication:} The system should have a user authentication system to prevent unauthorized 
    access.
    \item \textbf{Reports generation:} The system should be able to generate daily and end-of-stay reports for guests 
    in a comprehensible user friendly way.
    \item \textbf{Open API standard:} The API should comply with the Open API standard to ensure interoperability with 
    other systems.
    \item \textbf{Scalabitlity and maintainability:} The system should be able to handle a large number of sensors and 
    users. It should also be easy to maintain.
\end{enumerate}

The general system architecture of the proposed solution is shown in Figure \ref{fig:architecture-diagram}. The solution 
consists of four main layers: device, data, communication, and presentation. There are two main components, the smart
monitoring system and the web application that runs on the cloud. In the next sections, each layer and component will be
explained in detail.

\subsection{Device Layer}

\subsubsection{Microcontrollers}

It consists of three Espressif ESP32-DevKitC \cite{Espressif2024} microcontrollers (pictured in 
\ref{fig:connectivity-diagram}) connected to the sensors. One measures energy consumption, the other water usage and 
the third one indoor and outdoor temperatures. This development board was chosen for its extensive documentation, 
integrated WiFi and compatibility with the peripherals used. The idea was to have a low cost solution with fast 
development and deployment times.

FreeRTOS \cite{FreeRTOS2024} runs on top of these microcontrollers to allow a secure and resource aware solution for 
these embedded systems (microcontrollers). This was chosen as opposed to running the code on top of the chip because it 
allows to run multiple tasks at the same time with a predictible execution pattern, which is ideal for sensing and 
keeping connection with the internet alive at the same time (a requirement for this project).

The microcontrollers where programed in C++ using PlatformIO IDE (Integrated Development Environment) 
\cite{PlatformIO2024} for embedded systems. This development environment was chosen because of its compatibility with 
the microcontrollers and the sensors chosen, integration with thousands of libraries and collaboration principles.

\subsubsection{Sensors}

The interaction with the sensors was done using Arduino \cite{Arduino2024a} libraries, as these are the most used and 
well documented. They have been tested by a lot of projects (even in commercial products) and have an active community. 
The sensors used were the following:

\begin{itemize}
    \item \textbf{Energy usage:} Non-invasive AC Current Sensor SCT-013 \cite{CRCibernetica2024b} connected to 
    the ESP32 Energy Monitor Shield \cite{CRCibernetica2024c}.
    \item \textbf{Water consumption:} Liquid Flow Meter - Plastic 1/2" NPS Threaded TF-S201 \cite{CRCibernetica2024d}.
    \item \textbf{Temperature:} DS18B20 Temperature Sensor \cite{CRCibernetica2024e}.
\end{itemize}

Communication between the device layer and the data layer is done using the MQTT (Message Queuing Telemetry Transport) 
\cite{MQTT2024} protocol. MQTT was chosen because it is lightweight, secure and uses minimal network bandwidth. It also 
has become a standard in IoT for connecting remote devices.

\subsection{Data Layer}

Two separate databases are kept, a time series database for every reading from the sensors and a non-relational database 
for user and listing data, as well as device metadata. For the time series database the open source ThingsBoard IoT 
Platform \cite{ThingsBoard2024}was selected because it has integrated data collection and device management, which makes 
it ideal to collect data from sensor at big scale as opposed to an in house solution. The platform also has a well 
documented API for interaction with the devices and the data. 

\subsection{Communication Layer}

It is a REST API that interconnects the data layer with the application layer. It works as a way to have access to user, 
listing, and device (including time series) data. It complies with Open API \cite{OpenAPI2024} specification for 
standardization. This layer allows clients (applications) to access information from the data layer in a secure and 
controlled way.

\subsubsection{Communication with Airbnb}

Currently AirBnB does not have a public API, it provides a private API for authorized partners \cite{Airbnb2024c}. 
Becoming a partner was not possible for this project as there is no official way to apply for this. Rather, an AirBnB
manager contacts potential external partners. The solution was to simulate responses from the API using a web browser
to extract the payload from network activity on an AirBnB account using the web application.

\subsection{Presentation Layer (User Interface)}

A User Interface (UI) that can be used from personal computers, tablets and cellphones. It gives the host access to
a dashboard with in-depth real-time data of every listing that shows energy and water consumption, and temperature.

\section{Implementation}

\subsection{Sensing Nodes}

Three types of sensing nodes were developed for the project: power, water flow, and temperature. Each node is built on 
the ESP32 microcontroller and uses specific sensors to measure its respective parameters.

\subsubsection{Power Node}

The power node is the most complex of the three. It uses two current sensors connected to each phase of the 120V AC 
power line, with a small circuit built to adapt the current for measurement by the microcontroller's ADC (Analog to 
Digital Converter) input. This circuit includes a voltage divider to remove the negative part of the sine wave, a 
capacitor to stabilize the signal, and a resistor to convert the current to voltage. The resistor and capacitor values 
were calculated using Arduino's EmonLib documentation \cite{Arduino2024b}.

Initially, the ESP32 could interact with the sensors and retrieve data without issues, but as the full program was 
loaded, it encountered memory and CPU limitations. Various optimizations were attempted, such as reducing variable use, 
removing log messages, and simplifying tasks, but the problem persisted. Eventually, an Arduino Leonardo was added to 
the circuit to handle sensor data, which was then passed to the ESP32 via UART (Universal Asynchronous Receiver 
Transmitter). This solution resolved the issue, allowing the node to monitor without further problems. Figure 
\ref{fig:energy-node-schematic} shows the power node schematic.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.8\textwidth]{energy-node-schematic.png}
    \caption{Power node schematic}
    \label{fig:energy-node-schematic}
\end{figure}

After reading the current, EmonLib was used to calculate the power by multiplying the current by the fixed voltage 
(120V). Since data was collected every second, the energy was determined by summing the power readings over a minute and 
dividing by 60 to get watt-seconds or Joules (as shown in equation \ref{eq:energy-calculation}). The calculated energy 
was then sent to the cloud every minute.

\begin{equation}
    \label{eq:energy-calculation}
    \text{Energy} = \frac{1}{60} \sum_{i=1}^{60} (I_i \times V_i)
\end{equation}

\subsubsection{Water Flow Node}

This node uses a water flow sensor to measure the flow rate, which generates a pulse via the Hall effect each time water 
passes through. The program counts the number of pulses per second and multiplies this by a calibration factor (7.11 in 
this case) to calculate the flow rate, then divides by 1000 to determine the liters that flowed in that second. The 
sensor was connected to the ESP32, as shown in Figure \ref{fig:water-flow-node-schematic}.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.5\textwidth]{water-flow-node-schematic.png}
    \caption{Water flow node schematic}
    \label{fig:water-flow-node-schematic}
\end{figure}

The volume of water that passed in the last minute was calculated using equation \ref{eq:water-volume-calculation} and 
then sent to the database.

\begin{align}
    \label{eq:water-volume-calculation}
    \text{Volume} &{}= \sum_{i=1}^{60} Q_i \times \underbrace{\Delta t}_{=1\text{s}}\\
                  &{}= \sum_{i=1}^{60} Q_i
\end{align}

\subsubsection{Temperature Node}

The temperature sensors were connected to the ESP32, as shown in Figure \ref{fig:temp-node-schematic}. These sensors 
were managed using the OneWire library \cite{Arduino2024c}. The temperature was read every 2 seconds for 60 seconds and 
then divided by 30 to obtain the average temperature for the past minute. Equation \ref{eq:temperature-calculation}
shows the mathematical representation of this. Once the average temperature was calculated, it was sent to the database.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.4\textwidth]{temp-node-schematic.png}
    \caption{Temperature node schematic}
    \label{fig:temp-node-schematic}
\end{figure}

\begin{equation}
    \label{eq:temperature-calculation}
    \text{Average Temperature} = \frac{1}{30} \sum_{i=1}^{30} T_i
\end{equation}

\subsubsection{Timeseries Database}

ThingsBoard's internal PostgreSQL \cite{PostgreSQL2024} was used as the timeseries database for storing historical data 
from the sensing nodes. This database was selected due to its integration with the ThingsBoard platform and optimization

API.

\subsection{REST API}

This API was designed to manage users, providers, property listings, and devices for a hospitality platform like AirBnB, 
allowing users to list and manage properties. It enables seamless integration with third-party providers such as AirBnB, 
VRBO, and HomeAway, facilitating the management of property data from smart devices.

\subsubsection{Users}

The API includes a user management system. Users can register by providing essential information such as their username, 
email, and password. Upon successful registration, the system confirms that the user has been created. Once registered, 
users can log in using their credentials (username and password), which the system authenticates. If successful, the 
user receives an authentication token for subsequent requests to ensure security. Each user has a unique ID linked to 
their providers, listings, and jobs.

In addition to managing personal information, users can sync their property listings with third-party providers like 
AirBnB or HomeAway, ensuring that all listed properties are automatically updated in the system.

\subsubsection{Providers}

Providers are third-party platforms like AirBnB, VRBO \cite{VRBO2024}, and Booking.com \cite{Booking2024}, where 
properties are listed. The API allows users to create and manage these provider connections. When a user adds a new 
provider, the system stores the provider information and links it to the user.

\subsubsection{Listings}

Listings represent the properties that users manage on the platform. The API provides functionality to import listings; 
once imported, they can be viewed as a list or individually by their unique ID. Users can fetch all their listings to 
see which properties they are currently managing. Additionally, the system allows users to enable or disable listings, 
giving them full control over which properties are monitored.

\subsubsection{Reservations}

end-of-stay consumption reports. The API allows users to fetch all reservations for a listing, as well as individual 
reservations by their unique ID.

\subsubsection{Devices}

These devices are linked to specific listings and can be managed through the API. The API supports adding new devices 
by providing details such as the device type (e.g., energy meter), its unique identifier, and the token that 
authenticates the device with ThingsBoard. Users can retrieve a list of all devices associated with their listings. 
If a user wants detailed information about a specific device, they can fetch it using the device's unique ID. The API 
also supports updating device information or removing devices from the system when they are no longer needed.

\subsubsection{Reports}

Reports are generated daily and at the end of the guest's stay. The API provides endpoints for fetching these reports 
for specific listings.

\subsubsection{Agenda Jobs}

Two types of jobs were implemented: one that runs every two minutes to import listings and reservations from the 
provider, and another that runs daily at 2 PM to calculate the consumption for the last 24 hours or the total 
consumption.

\subsubsection{Schema Components}

The API employs specific schemas to define the structure of data objects exchanged between the client and server. For 
instance, the User schema specifies required fields such as username, email, and password for user registration. The 
Provider schema outlines the necessary fields for creating a provider, including the provider name and associated user 
ID. Similarly, the Listing schema describes the structure for properties, detailing attributes like title, location, 
and provider. Each device also adheres to a schema, which includes fields for the device type, unique ID, authentication 
token, and associated listing. This consistent structure facilitates integration and ensures data integrity.

Additionally, the API provides a standardized ApiResponse schema to maintain uniform error and success messages across 
various endpoints. Whether creating a user or fetching device details, the response messages clarify the status of the 
operation.

\subsection{Continuous Integration}

A continuous integration pipeline was established to automate the build and test processes. The pipeline consists of 
several stages, each assigned a specific task. It is triggered whenever changes are pushed to the repository, ensuring 
that the code remains up-to-date and thoroughly tested.

\subsubsection{Firmware}

The code is built using the PlatformIO build system, which compiles the firmware for the sensing nodes. Any errors in 
the code are logged, and the build will fail if issues are detected. An example run of the pipeline is shown in Figure 
\ref{fig:firmware-build}.

\subsubsection{REST API}

Similarly, the REST API is built, and unit and integration tests are conducted to verify that changes do not disrupt 
existing functionality. Figure \ref{fig:api-build} illustrates an example run of the pipeline.

\subsubsection{Development Environment}

The development environment was created using Docker Compose \cite{Docker2024}, allowing for easy setup and teardown. 
This environment included containers for MongoDB, ThingsBoard, NodeJS (API), and Appsmith. Docker Compose configured all 
the containers and services in a single file, enabling users to start and stop the environment with a single command.

\subsubsection{Deployment}

The complete solution was deployed on a virtual machine hosted on Google Cloud. This virtual machine was set up with 
Docker and Docker Compose to run the development environment, albeit with a slightly different configuration. The 
deployment process involved SSHing into the virtual machine, updating the repository, and building or rebuilding the 
containers. While the original plan was to automate this process with a CI/CD pipeline, time constraints prevented this 
implementation.
