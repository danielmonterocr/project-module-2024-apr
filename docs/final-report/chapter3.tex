\subsection{Hardware Testing}

Two types of tests were carried out on the hardware components: calibration and response time. The calibration tests 
consisted of comparing the readings from the sensors with readings with off-the-shelf instruments such as multimeters,
a graduated cylenrder and a thermometer. The response time tests consisted of measuring the time it took for the sensors
to respond to changes in the environment, this was crucial to ensure that the sensors were able to capture changes in
real-time, for example, turning on an appliance, filling up a glass of water or opening a window.

\subsubsection{Current Sensors}

The sensors were connected to a power strip together with the multimeter, an induction stove and a kettle were used to
generate changes in current consumption. Table \ref{tab:current-sensors-calibration} shows the results of the 
calibration tests for the current sensors. While the sensors were able to capture changes in current, there was a small 
deviation from the readings obtained with the multimeter. The standard deviation was calculated to be less than 0.1A, 
which was within the acceptable range for the application. Figure \ref{fig:using-kettle-and-ac} shows the change in 
power consumption when turning on a kettel (left), the idea was to able to capture rapid change in current consumption 
and according to this image the sensors were able to do so.

\subsubsection{Water Flow Sensor}

Two different tests were conducted on the water flow sensor, first the sensor was connected to a funnel and a liter of
water was then poured into the graduated cylinder at different speeds: slow, medium and fast. The second test was to 
capture the change in water flow when filling up the cylinder at high speed. Table 
\ref{tab:water-flow-sensor-calibration} shows the results of the first test, here a problem with slow water flow was
identify, this was due to the sensor not being able to capture the change in water flow at low speeds. A series of
improvements in the code were made, like separating the code in the different tasks, to be able to capture this more 
accuratly but it was not possible, it was concluded that a higher precision sensor was needed but it was not possible to
get one in time for the project.

Figure \ref{fig:water-flow-sensor-response} shows the response of the water flow sensor when filling up the container,
the sensor was able to capture the change in water flow in less than 10 seconds, which was within the acceptable range
when thinking about fast common tasks like filling up a glass of water, anything below this speed might not be captured
by the sensor, i.e. opening the faucet to get a little stream of water.

\subsubsection{Temperature Sensors}

Testing the temperature sensors required two different tests, first the sensors were compared to a thermometer to check
the accuracy of the readings in a span of 12 hours, the results were satisfactory as the standard deviation was less
than 0.5 degrees Celsius. The second test was to check the response time of the sensors, this was done by placing one
sensor in a glass full of ice and the other in a glass full of boiling water, Figure 
\ref{tab:temperature-sensors-calibration} shows the results of the calibration tests for the temperature sensors.

Figure \ref{fig:temp-sensor-response} shows the change in temperature when the sensors were submerged, once the
temperature reached the maximum and minimum values the sensors were swapped to check the response time, it was less than
a minute which was within the acceptable range for the application, there are no changes so drastic in temperature
inside or outside a room.

\subsection{Back-end Testing}

A set of tests were written to evaluate the performance of the back-end system. One particular detail about the unit,
integration and system tests is that the OpenAPI specification was used to verify the payloads and responses from the
API, this was done to ensure that the documentation was in line with the actual implementation of the API, with this
approach, if a new feature is added to the API, the Swagger documents need to be updated first, hence it works as a 
single source of truth.

\subsubsection{Unit Testing}

Unit tests were developed for every endpoint and utility function in the API. Mocha and Chai were used to write the
tests in JavaScript. A total of 100 unit tests were written in order to test payloads, responses and error handling. 
Figure \ref{fig:api-unit-tests-results} shows the results of the tests run locally.

\subsubsection{Integration Testing}

Figure \ref{fig:api-integration-tests-results} shows the results of a run of the integration tests, these tests were
used to verify the well functioning of the different components of the API. For example, a test creates a user, then 
logs in, it creates some listings, then it modifies this listings and finally deletes them along with the user. These
tests were also created using Mocha \cite{Mocha2024} and Chai \cite{Chai2024}.

\subsubsection{System Testing}

System tests were written using Postman \cite{Postman2024}, the idea was to test the API as a whole, this included the 
entire flow of creating a user, logging in, registering a provider, pulling the listings and reservations from AirBnB, 
simulating a stay and finally fetching the reports. An image showing these tests can be seen in TODO: Figure X.

\subsubsection{Regression Testing}

Unit and integration tests were run automatically using GitHub Actions every time code was pushed to the repository. An
email was received if the code failed to be compiled or any of the tests failed to run. Figure \ref{fig:api-build} shows
a successful build of the API.

The approach used to test the system was to follow Test-Driven Development (TDD) principles. This involved writing tests 
before writing the code to implement the functionality. Tests were written in a bottom-up fashion, starting with 
hardware testing and then moving up the system layers. The tests were incorporated into Continuous Integration (CI) 
pipelines using GitHub Actions \cite{GitHub2024} to ensure that the tests were run automatically whenever changes were 
made to the codebase.

As stated in the first section of this chapter, the objectives of the project were achieved. The system is able to
perform all basic functionalities. Some of the decisions taken in the design phase like TDD and CI were crucial to 
deliver a robust solution even though they made the development process slower. In terms of security in IoT devices, the 
use of Rust \cite{Rust2024} would have been a better choice, but duo to the lack of experience with the language and the 
small ecosystem of sensor libraries it was decided to use C++ and PlatformIO instead.

Calibration of the sensors took a lot of trial and error and this should be considered for any future work as it had 
an impact on the original schedule. Low cost sensors were sourced locally because of budget and time constraints, but 
using more precise sensors would have been a better choice, especially when talking about the water flow sensor.

The use of JavaScript and Node.js for the API was a good choice. The development process was faster due to 
extensive support of the language and the vast amount of libraries available. The use of Agenda for scheduling tasks was
an excellent choice, it's integration with MongoDB and the ease of use made it a great tool for the job. The use of 
OpenAPI and Swagger integrated into the code and the tests was a great decision as the documentation was written in a 
more organic way.

The User Interface was developed using low-code tools, this was changed from the original plan of using Flutter
\cite{Flutter2024} or React \cite{React2024} because of the time constraints. Appsmith proved to be a fast and easy to 
use tool, with the power of JavaScript underneath, in the end all the features were implemented in the span of 2 weeks.

Finally, deploying to Google Cloud was a good choice, having a dedicated virtual machine available 24/7 was crucial for
storing the data coming from the sensors. The use of Docker made the development both locally and remotely easier, as 
it was possible to spin up the containers on both environments with the same configuration.
